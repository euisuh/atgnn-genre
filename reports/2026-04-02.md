## What happened today

## Bugs encountered and fixed

### 1. RunPod SSH proxy rejects non-PTY connections
**Problem:** Standard `ssh user@ssh.runpod.io "command"` returns `Error: Your SSH client doesn't support PTY` and exits 0. RunPod's gateway requires a real terminal allocation.
**Fix:** Use `paramiko` with `invoke_shell()` to get a proper PTY session. All remote commands
this session were sent through a paramiko interactive shell.

---

### 2. `torch_geometric` not installed on pod
**Problem:** `from torch_geometric.nn import knn_graph` raises `ModuleNotFoundError` at import.
RunPod's PyTorch image ships torch/torchaudio but not PyG or its extensions.
**Fix:**
```bash
pip install torch_geometric
pip install torch_scatter torch_sparse torch_cluster torch_spline_conv \
    -f https://data.pyg.org/whl/torch-2.4.0+cu124.html
```
Pod has torch 2.4.1+cu124 — need the cu124 wheel index explicitly or you get a CPU-only build.

### 3. `wandb` not installed on pod

**Fix:** `pip install wandb -q`

### 4. `WANDB_API_KEY` not found — wandb login needed
**Problem:** Even after installing wandb and sourcing `.env`, running `python train.py` with
`--wandb_project $WANDB_PROJECT` failed with `UsageError: No API key configured`.
The env var was set (86 chars confirmed), but wandb didn't pick it up.
**Fix:** Explicit `wandb login $WANDB_API_KEY` to write credentials to `/root/.netrc`. After
that, all subsequent runs authenticated automatically.

---

### 5. `$WANDB_PROJECT` empty inside `nohup bash -c '...'` single quotes
**Problem:** Running the smoke test as:
```bash
nohup bash -c 'python train.py ... --wandb_project $WANDB_PROJECT ...' > log &
```
resulted in `error: argument --wandb_project: expected one argument`. Single quotes prevent
variable expansion, so `$WANDB_PROJECT` passed literally; inside the subshell it's undefined.
**Fix:** Write the command to a script file via base64 and run the file:
```bash
echo "<base64>" | base64 -d > /tmp/run_smoke.sh
nohup /tmp/run_smoke.sh > log &
```
The script sources `.env` with `set -a` before the Python call, so all variables expand correctly.

### 6. Multiple competing smoke test processes → CUDA OOM
**Problem:** After each failed attempt, the previous `train.py` process was not killed before
launching a new one. Three separate runs accumulated:
- PID 3326 (batch=24, started 02:54, 42 min CPU time)
- PID 3822 (batch=24, started 02:58, 38 min CPU time)
- PID 5043 (batch=8, started 03:10, 26 min CPU time)

All three competed for the 24 GB GPU. `nvidia-smi` showed 21.5 GB in use, 2.5 GB free —
not enough for even a small batch. The OOM error:
```
torch.OutOfMemoryError: CUDA out of memory. Tried to allocate 2.00 MiB.
GPU 0 has a total capacity of 23.54 GiB of which 1.12 MiB is free.
Process 2930104 has 9.07 GiB memory in use.
Process 2932560 has 9.07 GiB memory in use.
Process 2938599 has 5.38 GiB memory in use.
```
**Fix:** `pkill -9 -f train.py` to kill all competing processes. GPU recovered to 24 GB free
within seconds. Also added `PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True` to reduce
fragmentation.
**Prevention:** Always run `pkill -9 -f train.py` before re-launching any training.

## Critical performance issue discovered: real-time mp3 decoding

The smoke test has been running for **>34 minutes** on 2 epochs of 1986 tracks on an RTX 4090.
GPU utilization is only ~43%, confirming the bottleneck is **CPU-side data loading, not the GPU**.

### Root cause

`dataset.py` loads raw mp3 files at training time:
```python
def _load_audio(self, path):
    waveform, sr = torchaudio.load(path)          # mp3 decode — slow
    waveform = torchaudio.functional.resample(...)  # resample — slow
    ...
def _to_logmel(self, waveform):
    mel = self.mel_transform(waveform)             # mel spectrogram — slow
    ...
```

Per-file cost on CPU: ~0.5–2s for decode + resample + spectrogram.
With `num_workers=4`, `batch_size=16`: each batch takes ~4–8s of wall-clock data prep.
With ~125 batches/epoch × 2 epochs = 250+ batches, that's 16–33 minutes just for data loading.
The GPU sits idle most of this time waiting for the next batch.

### Fix: offline spectrogram preprocessing

Precompute mel spectrograms once and save as `.pt` tensors. At training time, just load the
tensor — no decode, no resample, no FFT.

**Step 1 — Add `scripts/precompute_spectrograms.py`:**
Iterate all audio files, compute log-mel spectrogram identically to `_to_logmel`, save as
`<audio_dir>/<chunk>/<track_stem>.spec.pt`. One-time cost: ~10–30 min for 20 chunks.

**Step 2 — Modify `dataset.__getitem__` to prefer cached spec:**
```python
spec_path = audio_path.replace('.mp3', '.spec.pt')
if os.path.exists(spec_path):
    spec = torch.load(spec_path, weights_only=True)  # fast
else:
    waveform = self._load_audio(audio_path)          # fallback
    spec = self._to_logmel(waveform)
```

**Expected impact:**
- Loading a `.pt` file: ~2–10ms (vs 500–2000ms for mp3 decode + mel)
- **100–200× faster data loading per sample**
- GPU utilization should jump from ~43% to ~95%+
- 2-epoch smoke test: minutes instead of hours
- 50-epoch Table 1 run: stays within budget instead of running all day

**Disk cost:** Each spectrogram `(1, 128, 1024)` float16 = ~262KB.
~50k tracks × 262KB ≈ ~13GB additional on the network volume (well within capacity).
Use float16 to halve storage; training casts to float32 on GPU anyway.
